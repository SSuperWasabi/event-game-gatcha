# 저장 시 재고 보존 — 설계 (2026-07-08, 현장 사고 대응)

## 사고

운영 2일차, 관리자 IP/상품 설정에서 수량 조정 후 저장 버튼 → **당일 실재고 카운트 전체 리셋**. 원인 두 겹:

1. 저장 버튼의 confirm("지금 재고를 초기화할까요?")에서 확인 터치 → `freshStock()` 전체 리셋 (휴먼 미스 유도 UI).
2. 잠복 결함: 등급 추가/삭제 후 저장 시 prizes·stock 배열 길이 불일치 → `reconcileStock`이 **해당 IP 재고 전체를 조용히 리셋**.

## 원칙 (사용자 확정)

- **저장 버튼 = 설정 저장만.** 재고(잔여 카운트)는 절대 건드리지 않는다. confirm 제거.
- **재고 전체 리셋 = 재고 탭 하단 버튼만** (기존 2단계 확인 유지).

## 구현

- `admSaveCfg`: confirm/리셋 분기 제거. 빈 등급 정리 시 stock을 같은 인덱스로 함께 정리(재고 보존). 저장 버튼 라벨 "💾 저장 (재고는 유지됨)" + 경고 안내문.
- **구조 변경 연산을 stock과 동기화 + 즉시 저장** (길이 불일치 → 전체 리셋 경로 원천 차단):
  - 등급 추가: stock에 새 칸 push. **새 등급은 `hidden:true`로 시작** — 이름 작성 전 참여자 노출/추첨 방지, 작성 후 노출 토글.
  - 등급 삭제: confirm 추가 + prizes·stock 같은 인덱스 splice (다른 등급 재고 유지).
  - 번들 전환: 해당 칸만 잔여 이어받아 형태 변환(해제=잔여 합계 숫자, 전환=[잔여]).
  - 하위 추가/삭제: 셀 배열 push/splice 동기화.
- `reconcileStock`은 부팅 백스톱으로 유지(정상 플로우에선 더 이상 발동 안 함).

## 검증 (헤드리스 실기)

7장 추첨(잔여 33) 상태에서: 이름·수량 수정+저장 → 재고 무변화 / 등급 추가(숨김·길이 동기화) / 빈 등급 정리 후에도 보존 / 삭제 시 해당 칸만 제거 / 재고 탭 리셋 정상. 예외 0. `APP_VER v25`, `sw.js kuji-v25`.
