## 1. 개요

### 1.1 배경 및 목적
- Com2uS 사내 팝업 스토어에서 **임직원 대상 무료 이벤트 게임**으로 운영한다. (구매·서베이 참가자 대상 무료 경품 추첨 성격)
- 일본 이치방쿠지 디지털 키오스크 UX를 레퍼런스로, **"정해진 수량에서 뽑는 복권식 가챠(쿠지)"** 경험을 제공한다.
- 자사 IP 굿즈를 경품으로 활용하여 임직원의 IP 친밀도와 이벤트 만족도를 높인다.
- 실제 결제가 아닌 무료 이벤트이므로, 기획·구현은 과한 뎁스 없이 **핵심 재미와 현장 운영 안정성**에 집중한다.

### 1.2 레퍼런스 분석 요약 (영상 기반)
실기 키오스크 플로우: ① 대기("화면을 터치하면 시작합니다") → ② **상품 시리즈(IP) 목록** (가격·잔여량 36/80 표시) → ③ 등급별 경품 상세(A상~J상, 등급별 잔여, LAST ONE상, DOUBLE CHANCE상) → ④ 참여 횟수 선택 → ⑤ 카드 결제.

본 프로젝트는 **④ 횟수 선택·⑤ 결제를 제거**(무료·1인 1회)하고, 영상에 없던 **뽑기 연출·결과 화면**을 추가한다. ②의 **IP 목록 선택은 그대로 채택**한다.

### 1.3 1차 프로토타입 범위
- **포함**: **멀티 IP(최대 3개)**, IP 선택 화면, 전체 키오스크 플로우(S1~S6), 비복원 추첨, **100% 당첨**, LAST ONE상, **앱 내 관리자 폼**, PWA 오프라인 동작, **디자인 시스템 적용(Lucky Draw 퍼플+골드, v1.4)**
- **제외(2차 이관)**: 결제, 사용자 인증/계정, 서버 연동, **DOUBLE CHANCE상**, **카메라 순간 촬영**, (로드맵) 다중 태블릿용 설정 백업·컨트롤타워

## 2. 타겟 & 운영 시나리오

### 2.1 운영 환경
- **기기**: 태블릿 1대 (iPad 또는 갤럭시탭, 세로 거치)
- **장소**: 사내 팝업 스토어 내 키오스크 거치대
- **운영**: 스태프 1인 이상 상주 (참여 통제 + 굿즈 지급 + 결과 화면 옆에서 확인)

### 2.2 현장 동선 (해피 패스)
1. 임직원이 줄을 서고, 스태프가 참여 자격 확인 (**1인 1회 — 스태프 수동 통제**)
2. 참여자가 키오스크 앞에 서면 대기 화면 터치 → IP 선택
3. IP 선택 → 경품 라인업 확인 → "뽑기 시작" → 티켓 선택 → 당첨 연출 (100% 당첨)
4. **결과 화면을 옆의 스태프가 확인** → 해당 등급 굿즈 지급
5. 스태프가 "처음으로" 버튼을 눌러 다음 참여자를 위해 대기 화면으로 복귀

### 2.3 엣지 시나리오
| 상황 | 처리 |
|---|---|
| 참여자가 결과 화면에서 이탈 (방치) | **30초** 무입력 시 자동으로 대기 화면 복귀 (당첨 로그는 이미 저장됨) |
| 특정 IP 티켓 소진 | 해당 IP를 IP 선택 화면에서 회색·"SOLD OUT" 처리, 선택 차단 |
| 전체 IP 소진 | 대기 화면에 "금일 이벤트가 종료되었습니다" 표시, 게임 진입 차단 |
| 태블릿 재시작/브라우저 종료 | localStorage·IndexedDB에 설정·재고·로그가 저장되어 이어서 운영 가능 |
| 행사장 와이파이 끊김 | PWA 오프라인 캐싱으로 정상 동작 (최초 1회 로딩만 온라인 필요) |
| 시스템 재고와 실물 재고 불일치 | 관리자 화면에서 당첨 로그 CSV로 대사(對査) |

## 3. 핵심 게임 메커니즘 — 비복원 추첨 (쿠지 방식)

> 일반 확률 가챠와 다름에 유의. **물리적 복권 상자를 시뮬레이션**한다.

### 3.1 규칙
1. **IP별로 독립된 티켓 풀**을 가진다. 각 IP의 전체 티켓 = 등급별 수량의 합 (예: A상 2 + B상 3 + C상 5 + D상 10 + E상 20 = 40장).
2. **100% 당첨**: 모든 티켓이 경품과 1:1로 대응한다. 빈 티켓(꽝)은 없다.
3. 참여자가 1장을 뽑으면 **잔여 티켓 중 균등 확률로 추첨**하고, 뽑힌 티켓은 그 IP 풀에서 **영구 제거**한다 (비복원).
4. 특정 등급이 소진되면 그 등급은 더 이상 나오지 않으며, 후반으로 갈수록 남은 등급의 체감 확률이 변한다.
5. **행운상**: 어떤 등급이든 **그 등급의 마지막 1개**를 뽑은 참여자에게 해당 경품 + **행운상(스토어 할인 쿠폰, 지류)을 현장 지급**한다. (등급별로 발동 — 한 회 이벤트에서 여러 번 나올 수 있음)
6. IP 풀이 비면 해당 IP는 SOLD OUT. 전체 IP가 비면 게임 진입 불가 (관리자 리셋 전까지).

### 3.2 구현 명세
- 추첨 로직: 선택된 IP의 잔여 수량 배열로 가중 풀을 만들고 `Math.random()`으로 1장 선택 → 해당 등급 수량 1 차감 → 즉시 localStorage 저장.
- **추첨 원자 처리 (연타 방지의 핵심)**: 티켓 터치 시 ①추첨 → ②재고 차감 → ③로그 저장 → ④입력 잠금 → ⑤연출 화면 전환을 **하나의 동기 동작**으로 처리. 잠금 이후의 추가 터치는 무시한다.
- 차감 시점은 **티켓 터치 직후**(연출 시작 전). 연출 중 새로고침해도 재고는 이미 차감된 상태여야 함 (중복 지급 방지가 미지급보다 우선).
- 당첨 로그: `{ timestamp(ISO), ipId, ipName, grade, prizeName, isLastOne, serial(당일 일련번호) }` 을 배열로 localStorage에 누적.
- **당일 세션 경계**: 일련번호와 통계는 **자정(로컬 기준)으로 구분**한다. 날짜가 바뀌면 일련번호는 1부터 다시 시작(로그 자체는 누적 보관).

## 8. Opus 구현 지시사항

### 8.1 베이스 코드
`kuji-event-game.html`을 **참고 베이스**로 사용한다. 화면 디자인 톤(보라 그라데이션, 노란 버튼, 다이아 로고, 티켓 연출)과 비복원 추첨 로직은 검증되었으니 재사용하되, 본 PRD와 다른 부분(IP 선택 화면 신설, 결제·횟수선택 제거→1인 1회 100% 당첨, **관리자 폼 설정**, PWA화, 30초, 스페셜 이펙트 조건)은 본 PRD를 따른다.

### 8.2 데이터 아키텍처 (외부 config.json 없음)
- **모든 설정은 앱 내 관리자 화면 폼으로 입력**한다. 별도 config.json 파일 편집은 하지 않는다.
- 저장 위치:
  - 설정(IP·등급·상품·수량·PIN·음소거) + 재고 + 당첨 로그 → **localStorage** (JSON 직렬화)
  - 업로드 이미지(경품·IP 배너·키비주얼) + 대기 영상 → **IndexedDB** (Blob/base64)
- 내부 데이터 스키마는 아래 형태(다중 IP). 관리자 폼이 이 구조를 읽고 쓴다.

```jsonc
{
  "eventName": "컴투스 팝업 쿠지",
  "adminPin": "0000",
  "idleTimeoutSec": 30,
  "muted": false,
  "npcs": [ { "id": "n…", "type": "image" } ],   // 뽑기 NPC 목록(이미지/영상). 미디어는 IndexedDB(npc_<id>)
  "ips": [                          // 최대 3개
    {
      "id": "main",
      "name": "서머너즈 워 컬렉션",
      "heroImageKey": "idb:hero_main",   // IndexedDB 키
      "lastOne": { "name": "스페셜 피규어", "imageKey": "idb:lastone_main" },
      "prizes": [                    // 등급 자유 구성 (A~J 등)
        { "grade": "A상", "name": "발키리 피규어", "imageKey": "idb:a", "count": 2,  "tier": "high"   },
        { "grade": "B상", "name": "이프리트 피규어", "imageKey": "idb:b", "count": 3,  "tier": "high"   },
        { "grade": "C상", "name": "빅 쿠션",        "imageKey": "idb:c", "count": 5,  "tier": "high"   },
        { "grade": "D상", "name": "아크릴 스탠드",   "imageKey": "idb:d", "count": 10, "tier": "normal" },
        { "grade": "E상", "name": "스티커 세트",     "imageKey": "idb:e", "count": 20, "tier": "normal" }
      ]
    }
  ]
}
```
- `imageKey`가 없거나 로드 실패 시 placeholder(이모지 또는 등급 텍스트)로 폴백 — 이미지 미준비 상태에서도 전체 플로우가 동작해야 함.
- **업로드 미디어 저장**: 경품/IP 이미지=dataURL, 대기영상·BGM·효과음·NPC=Blob 원본. 모두 IndexedDB. BGM/효과음은 업로드 시 그 파일을, 없으면 기본 합성음(효과음)·무음(BGM).
- **재고(remaining)는 설정과 분리해 저장**한다(설정 변경이 진행 중 재고를 덮어쓰지 않게). 관리자가 수량을 바꾸면 "재고를 초기화할까요?" 안내.
- `tier: "high"`는 스페셜 이펙트 대상(보통 A/B/C상). 스페셜 상품(LAST ONE)도 이펙트 대상.

### 8.3 파일 구조
```
kuji-app/
├── index.html            # 앱 본체 (화면·로직·관리자 폼, CSS/JS 인라인 허용)
├── manifest.webmanifest  # PWA 매니페스트 (세로 고정, fullscreen)
├── sw.js                 # service worker (자산 사전 캐싱, 캐시 버전 관리)
├── assets/
│   ├── icon-192.png       # PWA 아이콘
│   ├── icon-512.png
│   ├── bgm/               # 화면별 BGM (기본 제공분)
│   └── sfx/               # 효과음 (티켓·스페셜)
└── README.md             # 배포·운영 가이드 (8.4 포함)
```
> 경품/IP 이미지·대기 영상은 파일이 아니라 **관리자 화면에서 업로드 → IndexedDB**에 저장.

### 8.4 배포 & 운영 절차 (README에 포함)
1. 정적 호스팅(GitHub Pages 또는 Vercel)에 `kuji-app/` 업로드 (HTTPS 필수 — service worker 요건)
2. 태블릿 브라우저로 URL 접속 → 전체 로딩 확인 → "홈 화면에 추가"
3. 홈 화면 아이콘으로 실행 → **관리자 화면에서 IP·등급·수량·이미지·영상 입력** → 운영 시작
4. 콘텐츠 수정: 관리자 화면에서 편집 (재배포 불필요). 앱 자체 업데이트 시에만 온라인 새로고침으로 sw 캐시 갱신
5. 키오스크 잠금: iPad는 설정 > 손쉬운 사용 > 가이드 접근(Guided Access), 갤럭시탭은 앱 고정

### 8.5 테스트 체크리스트 (구현 완료 기준)
- [ ] 한 IP의 전체 티켓을 끝까지 뽑았을 때: 등급별 당첨 횟수 합 = 초기 수량과 정확히 일치 (100% 당첨, 비복원)
- [ ] 마지막 1장에서 LAST ONE상 연출 + 로그 `isLastOne: true`
- [ ] 티켓 연타/더블탭 시 추첨이 **1회만** 처리됨 (원자 락)
- [ ] IP별 SOLD OUT 표시·선택 차단, 전체 소진 시 진입 차단
- [ ] 뽑기 직후(연출 중) 새로고침 → 재고 차감·로그 존재
- [ ] 관리자: PIN 진입, IP/등급/상품/수량 편집, 이미지 업로드·표시, CSV 내용 검증, 리셋 후 초기 상태
- [ ] IP 1개 설정 시 S2 생략, 2~3개 시 S2 표시
- [ ] 30초 방치 시 S2/S3/S6에서 대기 화면 자동 복귀
- [ ] 스페셜 이펙트가 A/B/C/스페셜에서만 발생, 그 외 등급에선 미발생
- [ ] 자정 경계 후 일련번호 1부터 재시작
- [ ] 오프라인(비행기 모드) 재실행 → 전체 플로우 정상
- [ ] iPad Safari·갤럭시탭 Chrome 레이아웃·터치 동작 확인
